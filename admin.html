Full-stack architecture (target model)

Maneno ships today as a static client app (zero backend) — fastest to build and free to host. This admin manages content locally via a browser-stored overlay. Here is how it scales to a full stack when accounts & cross-device sync are needed:

            ┌────────────────────────────────────────────┐
 CLIENT     │  Maneno PWA  (index / play / register / admin)│
 (browser)  │  • offline-first service worker              │
            │  • localStorage: profile, settings, overlay  │
            └───────────────┬────────────────────────────┘
                            │ HTTPS / JSON
            ┌───────────────▼────────────────────────────┐
 EDGE/API   │  Vercel Functions (Fluid Compute, Node)      │
            │  /api/words  /api/profile  /api/daily        │
            │  /api/admin/*  (auth-gated CRUD)             │
            └───────────────┬────────────────────────────┘
                            │
            ┌───────────────▼───────────┐  ┌───────────────┐
 DATA       │  Postgres (Marketplace)    │  │  Auth         │
            │  words, users, scores,     │  │  (Sign in w/  │
            │  daily_seed, audit_log     │  │   Vercel/OAuth│
            └────────────────────────────┘  └───────────────┘

Admin (this page) → full stack: today writes a localStorage overlay; in the full model the same screens call /api/admin/words (create/update/hide) writing to Postgres, behind role-based auth. The game reads /api/words (cached at the edge) instead of the bundled words.js. Player profiles & streaks move from localStorage to the users table for cross-device sync. Nothing else in the UI changes — the data source swaps from a file to an API.
